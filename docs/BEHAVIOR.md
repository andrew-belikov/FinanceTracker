# Поведение FinanceTracker

Этот документ описывает фактическое поведение `tracker` и `bot` по текущему коду, включая источники данных, формулы и fallback-ветки.

## Доступ И Доставка Сообщений

- Бот отвечает только пользователям из `ALLOWED_USER_IDS`.
- Этот же набор используется как `TARGET_CHAT_IDS` для автоматических сообщений.
- Неавторизованные пользователи игнорируются молча.
- Отправка текста идёт через `safe_send_message(...)`: если Markdown ломается на динамических данных, бот повторяет ту же отправку без `parse_mode`.

## Как Проект Читает Данные

### Raw operations без dedup и без `state`-фильтра

Следующие сценарии читают `operations` в сыром виде:

- `/today`
- `/week`
- `/month`
- `/history`
- `/twr`
- триггер нового максимума
- триггер выполнения годового плана

Это означает:

- суммы пополнений и часть расходов здесь не фильтруются по `state = OPERATION_STATE_EXECUTED`;
- dedup по `operation_id` в этих ветках не выполняется.

### Dedup + только `OPERATION_STATE_EXECUTED`

Команда `/year` использует отдельный `OPERATIONS_DEDUP_CTE` и фильтр `state = OPERATION_STATE_EXECUTED`.

Практическое следствие: цифры `/year` могут отличаться от `/month`, `/week` и `/history`, если в `operations` есть дубликаты, промежуточные состояния или неисполненные записи.

### Зависимость от `income_events`

`income_events` используется для:

- купонов и дивидендов в `/today`, `/week`, `/month`;
- части налогов в `/today`, `/week`, `/month`;
- минутных уведомлений о зачислении дохода.

Если таблицы `income_events` нет:

- купоны и дивиденды для этих отчётов становятся `0`;
- налоги считаются только по налоговым типам операций из `operations`;
- минутные уведомления о купонах и дивидендах не работают.

## Команды Бота

### `/start`

- Доступна только авторизованным пользователям.
- Возвращает короткое приветствие и отправляет к `/help`.

### `/help`

- Показывает список команд и фоновых задач.
- Текст подсказки человекочитаемый, но фактическое расписание ежедневного JobQueue определяется локальным временем контейнера.

### `/today`

Источник данных:

- последние 2 записи из `portfolio_snapshots`;
- сумма всех пополнений из `operations` по типу `OPERATION_TYPE_INPUT`;
- внешние пополнения и выводы из `operations` за локальный день `TIMEZONE` для корректировки дневной дельты;
- купоны и дивиденды из `income_events` за локальный день `TIMEZONE`;
- комиссии и налоги из `operations` и `income_events`.

Как считаются ключевые поля:

- `snapshot_dt` это `snapshot_at`, конвертированный в `TIMEZONE`;
- дельта за день = `последний total_value - предыдущий total_value - net_external_flow_day`, где `net_external_flow_day = deposits - withdrawals`;
- общий PnL = `последний total_value - сумма всех пополнений`.

Поведение:

- если снапшотов нет, возвращается `Пока нет ни одного снапшота портфеля.`;
- если снапшот один, дневная дельта не считается;
- выводы исключаются из дневной дельты, но отдельной строкой не показываются;
- текст выбирается случайно из набора шаблонов в [`src/bot/today_templates.py`](../src/bot/today_templates.py), поэтому формулировка может меняться при одинаковых числах.

### `/week`

Период:

- текущая рабочая неделя понедельник-пятница, определённая в `TIMEZONE`.

Источник данных:

- последний снапшот как текущая стоимость;
- последний снапшот строго до начала недели как база для сравнения;
- пополнения, выводы, купоны, дивиденды, комиссии и налоги за период текущей рабочей недели;
- пополнения с начала года для прогресса плана.

Как считаются ключевые поля:

- недельная дельта = `current_value - last_snapshot_before_monday - net_external_flow_week`, где `net_external_flow_week = deposits - withdrawals`;
- недельный процент = `delta / base`;
- прогресс плана = `пополнения с 1 января / PLAN_ANNUAL_CONTRIB_RUB`.

Fallback:

- если снапшота до начала недели нет, бот считает, что портфель появился внутри недели: `week_delta_abs = current_value - net_external_flow_week`, `week_delta_pct = 0.0`;
- если снапшотов нет вообще, возвращается `Пока нет ни одного снапшота портфеля.`;
- выводы исключаются из недельной дельты, но отдельной строкой не показываются;
- текст выбирается случайно из [`src/bot/week_templates.py`](../src/bot/week_templates.py).

### `/month`

Период:

- текущий календарный месяц в `TIMEZONE`.

Источник данных:

- пополнения, выводы, купоны, дивиденды, комиссии и налоги за текущий месяц;
- пополнения YTD для годового плана;
- последний снапшот строго до начала месяца и последний снапшот внутри месяца;
- позиции этих двух снапшотов для блока изменений.

Как считаются ключевые поля:

- изменение за месяц = `end_snap.total_value - start_snap.total_value - net_external_flow_month`, где `net_external_flow_month = deposits - withdrawals`;
- прогресс годового плана = `dep_year / plan`;
- фраза о статусе графика выбирается случайно по сравнению `dep_year` с линейным ориентиром на текущую дату:
  - `>= 105%` ориентира: впереди графика;
  - `>= 95%` ориентира: около графика;
  - иначе: позади графика.

Блок изменений позиций:

- строится по `figi` между начальным и конечным снапшотом месяца;
- категории: новая позиция, закрыта, увеличена, уменьшена;
- сортировка внутри категории идёт по тикеру.

Fallback:

- если нет стартового снапшота до начала месяца, изменение не считается;
- если нет конечного снапшота в месяце, текущая стоимость остаётся `0`;
- выводы исключаются из месячной дельты, но отдельной строкой не показываются;
- текст выбирается случайно из [`src/bot/month_templates.py`](../src/bot/month_templates.py).

### `/year`

Аргументы:

- без аргумента: текущий год YTD;
- `/year YYYY`: указанный календарный год.

Источник данных:

- снапшоты начала и конца периода;
- dedup-слой `operations_dedup`;
- только операции со `state = OPERATION_STATE_EXECUTED`;
- позиции первого и последнего снапшота внутри периода;
- `income_events` здесь не используются, расчёты идут прямо по `operations`.

Что отправляет команда:

1. summary-текст;
2. PNG-график `стоимость портфеля и пополнения`, если данных достаточно;
3. PNG-график `прирост/падение по месяцам`, если данных достаточно;
4. текстовый блок изменений позиций.

То есть при нормальных данных `/year` обычно отправляет 4 сообщения, но при недостатке данных графиков может быть меньше.

Как считаются ключевые блоки summary:

- стоимость на конец периода = последний снапшот внутри периода;
- изменение стоимости = `end_snap.total_value - start_snap.total_value`, где `start_snap` это последний снапшот строго до начала периода;
- прогресс плана = `YTD deposits / PLAN_ANNUAL_CONTRIB_RUB`;
- `Реализовано` по активам = сумма `yield + commission` по продажам `OPERATION_TYPE_SELL`;
- `Дивиденды/купоны` по активам = сумма дивидендов, купонов и соответствующих налоговых строк;
- `По открытым позициям` = сумма `expected_yield` последнего снапшота на конец периода.

Как строится блок изменений позиций:

- берутся первый и последний снапшот внутри периода;
- позиции агрегируются по "логическому активу" в порядке `ticker -> name -> figi`;
- валютные позиции, включая `RUB000UTSTOM` и `instrument_type=currency`, исключаются;
- категории: `🆕 Новые`, `📈 Докупили`, `📉 Продали часть`, `✅ Закрыли`;
- блок `🆕 Новые` показывается только если период начинается 1 января.

Fallback:

- если внутри периода меньше двух снапшотов, вместо diff-блока возвращается сообщение о нехватке данных;
- если нет данных для первого графика, бот отправляет текст `Недостаточно данных для графика ...`;
- если нет данных для второго графика, второй PNG просто не отправляется.

### `/structure`

- Берёт последний снапшот и его позиции.
- Группирует позиции по типам инструментов: акции, облигации, ETF, валюта, фьючерсы, другое.
- Для каждой группы показывает сумму, долю портфеля и совокупный P&L.
- Затем перечисляет бумаги внутри группы по убыванию стоимости позиции.

Fallback:

- если снапшотов нет, сообщение `Нет ни одного снапшота портфеля.`;
- если в снапшоте нет позиций, сообщение `В последнем снапшоте нет позиций.`

### `/history`

- Строит PNG-график стоимости портфеля и кумулятивных пополнений.
- Берёт `portfolio_snapshots` как временной ряд и `operations` как источник пополнений.
- Ось X строится недельным шагом от первой до последней даты снапшота.
- Для каждой точки графика берётся последний снапшот `<= date` и сумма всех пополнений `<= date`.

Fallback:

- если снапшотов меньше двух, бот отвечает `Недостаточно данных для построения графика.`

### `/twr`

- Агрегирует `portfolio_snapshots` по `snapshot_date`.
- Читает пополнения из `operations`, переводит naive UTC timestamp в локальную дату `TIMEZONE`.
- Считает TWR посуточно по формуле `(V - CF) / V_prev - 1`.
- Отправляет короткий текст с текущим TWR и стоимостью портфеля, затем PNG-график.

Fallback:

- если исторических точек меньше двух, бот отвечает `Недостаточно данных`.
- если `V_prev` отсутствует, нулевой или текущая точка `None`, формула для этой точки не пересчитывается, а серия продолжает последнее известное значение.

## Автоматические Сообщения

### Ежедневный JobQueue

`daily_job(...)` выполняется каждый день в `18:00` по локальному времени контейнера.

Если бот стартовал позже этого времени, одноразовый startup catch-up добирает пропущенный daily job за текущую дату.

Он может отправить:

- monthly PDF, если сегодня последний день месяца;
- недельный отчёт, если сегодня пятница;
- один или несколько trigger-сообщений.

Если на конкретную дату отправлять нечего, джоб завершается тихо.
Повторный запуск за ту же дату подавляется через таблицу `bot_daily_job_runs`.

Month-end delivery policy:

- primary monthly artifact для auto-send — PDF через внутренний `reporter`;
- если `reporter` недоступен, PDF не собрался или `Ollama` не прошла валидацию narrative, бот отправляет текстовый `month-end fallback`;
- monthly PDF auto-send использует отдельный `job_name=monthly_pdf_delivery`, поэтому может ретраиться независимо от `daily_summary`;
- startup catch-up может добрать `monthly_pdf_delivery`, даже если `daily_summary` за ту же дату уже был закрыт ранее.

### Триггер Нового Максимума

Условие:

- текущая стоимость последнего снапшота строго больше максимума всех `total_value` до этой даты.

Особенность:

- для самого первого дня trigger не отправляется, потому что исторического максимума "до этого дня" ещё нет.

### Триггер Выполнения Годового Плана

Условие:

- сумма пополнений с 1 января до вчера была меньше плана;
- сумма пополнений с 1 января до сегодня стала больше или равна плану.

Источник данных:

- raw `operations` по типу `OPERATION_TYPE_INPUT`, без dedup и без `state`-фильтра.

### Уведомления О Купонах И Дивидендах

`check_income_events(...)` запускается каждые 60 секунд, первый раз через 10 секунд после старта.

Логика:

1. Читает все `income_events` с `notified = false`.
2. Для каждой строки отправляет отдельное сообщение во все `TARGET_CHAT_IDS`.
3. Помечает событие как `notified = true` только если отправка успешно прошла во все чаты.

Формат:

- купон: `💸`
- дивиденды: `💰`
- в тексте есть имя инструмента, тип события, сумма и доходность к cost basis.

## Поведение `tracker`

### Снапшоты

- В каждый локальный день `SCHED_TZ` хранится один снапшот на счёт.
- Повторный запуск в этот же день обновляет агрегаты и полностью пересоздаёт позиции снапшота.
- Если для позиции в БД нет `ticker` или `name`, `tracker` дёргает `GetInstrumentBy` и дозаполняет `instruments`.

### Операции

- Синхронизация использует `GetOperationsByCursor` с `withoutTrades=true`.
- Запрос постраничный, курсорный.
- `OPERATIONS_MAX_PAGES` защищает от бесконечной пагинации.
- Если курсор API повторился, `tracker` завершает синк с предупреждением.

### Backfill Полей `OperationItem`

Если в таблице `operations` у существующих строк для счёта есть `state IS NULL`, `tracker` считает схему неполной и запускает backfill от даты открытия счёта, а не от последней известной операции.

### Формирование `income_events`

- `OPERATION_TYPE_COUPON` и `OPERATION_TYPE_DIVIDEND` считаются gross.
- `OPERATION_TYPE_COUPON_TAX` и `OPERATION_TYPE_DIVIDEND_TAX` считаются tax.
- `net_amount = gross + tax`.
- `net_yield_pct = net_amount / latest_cost_basis * 100`, если cost basis положительный.
- Если `net_amount <= 0`, событие не создаётся.

## Известные Нюансы

- `/today`, `/week`, `/month` и `/history` могут не совпадать с `/year` из-за разных правил по dedup и `state`.
- `TIMEZONE` не сдвигает ежедневную отправку JobQueue.
- На чистой БД без SQL-миграции `deposits` остаётся таблицей, хотя новый код уже работает через `operations`.
- Формулировки `/today`, `/week` и `/month` недетерминированы из-за случайного выбора шаблона; числовая часть при этом берётся из одних и тех же расчётов.
