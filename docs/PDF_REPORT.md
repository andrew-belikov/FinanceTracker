# PDF Report Spec

Документ фиксирует канонический состав PDF-отчёта для `FinanceTracker`.

Связанный технический дизайн:

- [PDF_REPORT_TECH.md](./PDF_REPORT_TECH.md)

Первая целевая версия: месячный обзор портфеля.

## Цель

PDF должен быть:

- читаемым как самостоятельный месячный обзор без Telegram;
- опираться на детерминированные расчёты проекта, а не на LLM;
- собираться из HTML-шаблона и печататься в PDF без ручной верстки;
- быть пригодным и для личного чтения, и для будущего web/print режима.

## Базовые правила

- Все денежные расчёты, сортировки и агрегаты делает код проекта.
- `Ollama` или любая другая модель пишет только narrative-блоки:
  - заголовок;
  - executive summary;
  - краткие комментарии к графикам;
  - риски и наблюдения.
- Модель не должна:
  - считать `PnL`, `TWR`, `cashflow`, `rebalance`;
  - выбирать итоговые цифры страницы;
  - быть единственным источником правды для PDF.
- Отчёт должен работать даже без AI-слоя.

## Общие принципы верстки

- Формат: `A4 portrait`.
- Стиль: спокойный `family office review`, а не брокерская выписка.
- Язык интерфейса: русский; английские термины допустимы только там, где без них теряется общепринятый смысл метрики, например `TWR`.
- На странице один главный вопрос и один доминирующий визуальный блок.
- Не использовать мозаику из карточек как основной паттерн.
- Один крупный график на страницу лучше, чем несколько мелких.
- Таблицы должны быть плотными, но легко читаемыми на печати.
- Числа, даты и тикеры не должны разваливаться на случайные переносы.
- Первая страница не должна оставаться полупустой: hero-блок, ключевые метрики и факты месяца занимают основную высоту листа.
- Графики должны содержать подписи ключевых точек: последняя точка, экстремумы месяца, лучшие/худшие дни.
- Цвета должны наследовать палитру текущих графиков:
  - portfolio
  - deposits
  - twr
  - positive
  - negative

## Monthly Review

Канонический monthly PDF состоит из `5` страниц.

### Страница 1. Overview

Задача страницы:

- быстро ответить, чем месяц закончился;
- показать несколько ключевых цифр;
- дать короткий narrative summary.

Наполнение:

- заголовок периода:
  - `апрель 2026`
- служебный блок:
  - счёт;
  - период;
  - таймзона;
  - дата генерации;
- главная цифра:
  - стоимость портфеля на конец периода;
- ключевые метрики:
  - результат месяца в рублях;
  - результат месяца в процентах;
  - пополнения за месяц;
  - пополнения YTD;
  - купоны/дивиденды за месяц;
  - комиссии;
  - налоги;
  - `reconciliation_gap_abs`;
- executive summary:
  - 3-4 коротких пункта;
- блок фактов:
  - лучший день месяца;
  - худший день месяца;
  - количество income events;
  - top holding.

Расположение:

- слева крупный заголовок и главная цифра;
- справа компактная summary-колонка;
- ниже hero-числа — полоса из ключевых метрик;
- нижняя часть страницы занята блоком фактов месяца и блоком notes/signal summary.

Текст:

- short summary на 80-120 слов;
- без рекламного тона;
- без “магических” выводов;
- только интерпретация уже посчитанных цифр.

### Страница 2. Performance

Задача страницы:

- показать, как месяц шёл во времени;
- отделить рыночную динамику от внешних движений денег.

Наполнение:

- один главный график на большую часть страницы.

Рекомендуемая композиция:

- верхняя панель:
  - `portfolio value` по дням месяца;
- нижняя панель:
  - `net cashflow` по дням;
  - `day_pnl` по дням;
- справа короткий commentary block:
  - лучший день;
  - худший день;
  - дата локального максимума;
  - дата локального минимума;
  - 3-4 bullet points по ходу месяца.

Подписи на графике:

- последняя точка стоимости;
- локальный максимум и локальный минимум по стоимости;
- лучший и худший день по `day_pnl`.

Что не делать:

- не лепить отдельную тяжёлую legend-box;
- не смешивать слишком много серий в одну ось;
- не использовать intraday-формулировки: график опирается на end-of-day данные.

### Страница 3. Structure & Current Positions

Задача страницы:

- показать, чем портфель является на конец месяца;
- зафиксировать текущую структуру и концентрацию.

Наполнение:

- блок структуры по классам активов:
  - акции;
  - облигации;
  - ETF;
  - валюта;
  - фьючерсы;
- визуал структуры:
  - donut или компактный stacked bar;
- таблица `top positions`:
  - инструмент;
  - стоимость позиции;
  - вес;
  - `expected_yield`;
  - `expected_yield_pct`.

Расположение:

- слева визуал структуры;
- справа таблица `top positions`;
- внизу короткий текст:
  - где концентрация;
  - что является главным балластом;
  - что является главным драйвером по весу.

Примечание:

- эта страница отвечает на вопрос “что в портфеле сейчас”, а не “что произошло внутри месяца”.

### Страница 4. Instruments Deep Dive

Это расширенная инструментальная страница.

Именно здесь должны появиться блоки:

- что зашло;
- что вышло;
- что увеличилось;
- что уменьшилось;
- что сильнее всего росло;
- что сильнее всего просаживалось;
- что дало наибольший вклад в результат;
- какие дивиденды и купоны пришли.

Задача страницы:

- превратить “месяц по портфелю” в “месяц по инструментам”.

Страница делится на `3` смысловых блока.

Правило компоновки:

- не собирать три узкие таблицы в один ряд;
- prefer `2-column` layout с отдельным блоком movers и отдельным блоком contribution;
- в таблицах объединять тикер и название в один столбец `Актив`, чтобы снизить горизонтальный шум.

#### Блок A. Position Flow

Задача:

- показать изменения состава между началом и концом периода.

Наполнение:

- четыре группы:
  - `Новые`;
  - `Закрытые`;
  - `Докупили`;
  - `Сократили`.

Источник:

- сравнение позиций первого и последнего релевантного snapshot в периоде.

Правило группировки:

- использовать `logical_asset_id` как primary key для PDF-версии;
- fallback допустим в порядке:
  - `asset_uid`
  - `instrument_uid`
  - `figi`
  - `ticker`

Поля в строке:

- название инструмента;
- ticker;
- количество на начало;
- количество на конец;
- изменение количества;
- при наличии:
  - изменение рыночной стоимости позиции.

Расположение:

- 2 колонки:
  - слева `Новые` + `Закрытые`;
  - справа `Докупили` + `Сократили`.

#### Блок B. Intramonth Movers

Задача:

- показать, какие инструменты сильнее всего росли и просаживались внутри месяца.

Важно:

- формулировка должна быть честной:
  - не `intraday`;
  - не “рыночная свеча”;
  - а именно `end-of-day trajectory within month`.

Минимальный целевой набор метрик по инструменту:

- `eod_min_position_value`
- `eod_max_position_value`
- `eod_end_position_value`
- `eod_min_expected_yield`
- `eod_max_expected_yield`
- `eod_end_expected_yield`
- даты минимума и максимума

Ранжирования для страницы:

- `Top growth during month`
  - по наибольшему росту `expected_yield` или `position_value` между локальным минимумом и последующим максимумом;
- `Top drawdown during month`
  - по наибольшему падению `expected_yield` или `position_value` между локальным максимумом и последующим минимумом.

Для monthly MVP рекомендуется использовать:

- основной рейтинг:
  - по `expected_yield` в рублях;
- вторичная колонка:
  - `expected_yield_pct`.

Почему так:

- абсолютный вклад понятнее в PDF;
- процент нужен как вторичная нормализация.

Ограничение:

- метрика строится по `portfolio_positions` из snapshot’ов;
- это `EOD-only`, а не intraday truth;
- на неё влияют сделки внутри месяца;
- это нужно явно указать мелким пояснением под таблицей.

Расположение:

- две таблицы рядом:
  - `Сильнее всего росло`;
  - `Сильнее всего просаживалось`.

#### Блок C. Contribution To Result

Задача:

- ответить, какие инструменты реально дали прибыль/убыток и какие income пришли.

Блок делится на три подблока.

1. `Realized contribution`

Источник:

- операции продажи.

Метрика:

- `realized_net_by_asset = sum(yield + commission)` по `sell`-операциям за период.

Поля:

- инструмент;
- ticker;
- realized net.

2. `Income contribution`

Источник:

- `operations` и/или `income_events`.

Метрика:

- `income_net_by_asset`
  - дивиденды;
  - купоны;
  - связанные налоговые строки.

Поля:

- инструмент;
- тип income event;
- net amount.

3. `Open P/L at period end`

Источник:

- последний snapshot периода.

Метрика:

- `expected_yield` на конец периода.

Поля:

- инструмент;
- open P/L;
- open P/L %.

Важно:

- `Open P/L at period end` не равно месячному вкладу в результат;
- это snapshot-state на конец периода;
- в PDF это нужно подписывать отдельно, не смешивая с realized contribution.

Расположение:

- три компактных ranked lists:
  - `Реализовано`;
  - `Дивиденды/купоны`;
  - `Открытый P/L на конец месяца`.

### Страница 5. Cashflows & Data Quality

Задача страницы:

- показать крупные денежные движения;
- закрыть вопрос доверия к данным;
- дать appendix для проверки.

Наполнение:

- таблица крупнейших операций месяца:
  - дата;
  - тип;
  - инструмент/описание;
  - сумма;
- таблица income events;
- блок data quality:
  - `unknown_operation_group_count`;
  - `mojibake_detected_count`;
  - `positions_missing_label_count`;
  - `has_full_history_from_zero`;
  - `income_events_available`;
- таблица `reconciliation_by_asset_type`;
- appendix:
  - `generated_at`;
  - `timezone`;
  - `period_start`;
  - `period_end`;
  - `positions_value_sum`;
  - `reconciliation_gap_abs`.

## Narrative Blocks From Ollama

Для monthly PDF AI может писать только такие блоки:

- `report_title`
- `executive_summary`
- `performance_commentary`
- `instrument_takeaways`
- `risk_notes`

Все эти блоки должны приходить как structured JSON.

Пример допустимых задач для модели:

- объяснить, что в месяце было главным драйвером;
- кратко описать ротацию инструментов;
- выделить 2-3 риска или аномалии;
- подписать смысл графика человеческим языком.

Пример недопустимых задач:

- посчитать `PnL`;
- решить, какой инструмент дал biggest loss, если это не посчитал код;
- самостоятельно агрегировать сырые операции вместо подготовленного payload.

## Fallback Rules

Если данных недостаточно:

- страница не должна ломаться;
- блок заменяется короткой explanatory note.

Примеры:

- нет хотя бы двух snapshot’ов в месяце:
  - скрыть `Position Flow`;
- нет достаточного EOD-ряда по инструментам:
  - скрыть `Intramonth Movers`;
- нет income events:
  - показать пустой income state без ошибки;
- нет rebalance targets:
  - скрыть rebalance subsection, а не всю страницу.

## Не делать в первой версии

- не строить intraday chart;
- не делать “AI verdict score”;
- не смешивать monthly contribution и end-of-period open P/L в одну таблицу;
- не пытаться вытащить всё в один двухстраничный PDF;
- не делать отдельный дизайн под каждый месяц.
